# Open Source AI Project Explorer

## 核心目标

让初学者依次理解：

1. 项目解决什么问题，整体怎样运转；
2. 一次请求怎样穿过模型、Agent、工具和状态层；
3. 关键概念为何诞生、本质逻辑和上下游是什么；
4. 业务动作如何映射到项目里的关键函数、变量、配置、数据结构和事件；
5. 项目如何实现，各阶段输入输出、代码路径和设计边界是什么；
6. 哪些机制可以迁移，哪些只是当前项目的特殊选择。

最终文档必须同时满足两类读者：

- **先理解业务逻辑的读者**：能沿着真实场景读懂“谁触发、谁负责、谁消费、什么时候结束、失败归谁”；
- **准备读源码的读者**：能在每个关键业务动作后找到对应函数名、变量名、参数示例、事件名、状态字段或最小代码片段。

禁止两种极端：不能只写业务故事而没有源码落点，也不能堆砌函数和文件名导致读者看不懂业务流程。

分析结果必须持久化，使后续追问从索引和已有主题继续，而不是重新读取整个仓库。

## 任务模式

### 整体分析

首次接触项目时生成：

- `projects/<project>/00-overview.md`
- `projects/<project>/01-index.json`
- `projects/<project>/diagrams/architecture.mmd`

只建立整体设计思路、主流程和候选特性，不提前深挖所有能力。

### 特性深潜

用户追问具体设计时按需生成：

- `projects/<project>/topics/<feature>.md`
- `projects/<project>/prompts/<feature>-*.md`（仅当 Prompt 关键）
- 特性图源
- 更新后的 `01-index.json`

不是每个项目都必须分析 Memory，也不是每个特性都需要 Prompt 档案。

## 强制执行顺序

1. 确认源码 URL 或本地路径、输出位置和任务模式。
2. 检查 `projects/<project>/01-index.json`。
3. 索引存在时先读索引；整体问题再读 overview，特性问题再读对应 topic。
4. 只有现有文档不足时才进入源码。
5. 源码探索从 anchors 开始，证据足够后立即停止。
6. 生成或更新文档，并回写索引、版本和 FAQ 路由。
7. 验证 JSON、链接、Mermaid、章节锚点和源码证据。

源码获取、首次扫描和刷新规则见 [references/workflow.md](references/workflow.md)。

## 增量读取协议

这是不可跳过的约束：

- **索引优先**：后续追问先读 `01-index.json`。
- **已有主题优先**：命中 `topics` 或 `faq_routes` 时先读 topic。
- **锚点优先**：实现问题通常只读 2–6 个关键文件。
- **定向扩展**：新主题通常探索 5–15 个相关文件。
- **禁止习惯性重扫**：仅在索引缺失、版本显著变化或锚点失效时扩大范围。
- **局部刷新**：源码变化时按变更路径和 anchors 刷新受影响 topic。
- **持续回写**：发现更好的路径、符号和关键词后更新索引。

详细 Schema 和 Token 控制见 [references/indexing.md](references/indexing.md)。

## 整体导读要求

`00-overview.md` 按“学习目标 → 一句话定位 → 真实场景 → 整体设计思路 → 核心概念 → 请求生命周期 → 代码导读 → 候选特性 → 阅读路径”展开。

不要使用模块清单开篇，不要用连续表格代替解释。完整路径留给索引。

可从 [templates/overview.md](templates/overview.md) 开始。

## 特性深潜要求

### 第一部分：概念本质

进入源码前必须回答：

1. **为何诞生**：前身是什么，旧方案遇到什么边界？
2. **核心逻辑**：能否用一句“输入 → 变换 → 输出”概括？
3. **上下游**：属于哪个大类，上游输入、下游消费者和常见分支是什么？

分类按“解决的问题 → 例子 → 边界”介绍，全部讲完后才映射到代码。本阶段不要求读者背函数、配置、公式或操作步骤。

### 第二部分：项目实现

先映射通用概念，再讲流程。实现篇必须采用“业务解释 → 实现映射 → 示例 → 边界”的节奏，避免直接从函数清单开始。

每个 Phase、Stage 或读取层必须写清：

- 定位；
- 单独解决的问题；
- 输入；
- 关键处理；
- 输出和下游；
- 不负责什么；
- 并行、串行、分层或独立存在的原因。

每个关键业务动作还必须尽量补齐：

- **业务含义**：它在真实任务里对应什么动作，例如派工、检索、压缩、审批、回填、通知；
- **关键实现名**：涉及的函数、结构体、变量、配置、工具名、事件名或状态字段；
- **最小示例**：可读的 JSON、Prompt 片段、函数参数、消息内容、状态流转或伪代码；
- **输入输出关系**：上游给什么，处理后下游拿到什么；
- **失败和边界**：错误如何表达，谁重试，什么情况不应该用这个机制。

示例要服务理解，不要变成大段源码粘贴。优先使用短 JSON、短消息、短参数对象、短伪代码或 5–15 行以内的关键片段。

存在 summary、handbook、detail、raw evidence 等分层产物时，解释粒度、成本、精度和进入下一层的条件，以及为何不能只保留或读取一层。

### 第三部分：迁移应用

抽象出跨框架闭环，说明可迁移原则、不可照搬实现、适用场景、最小方案、成功指标、过期、纠错、隐私和删除。

### 第四部分：复习

用三句话复述问题、本质和上下游，列出误解、后续问题和证据。

使用 [templates/topic.md](templates/topic.md)。完整规范见 [references/writing-guide.md](references/writing-guide.md)。

## Prompt 分析

仅当 Prompt 真正控制特性行为时建档。说明行为问题、调用位置、结构区块、关键原文、行为影响、变量、解析器和下游消费者。引用关键片段，完整原文指向源码。

使用 [templates/prompt.md](templates/prompt.md)。

## 可视化要求

- 总览通常使用一张架构图和一张必要的主流程图；
- 特性使用概念地图、处理流程或时序图；
- 每张图只表达一个核心关系；
- 图后必须解释主路径和关键分支；
- Mermaid 节点 ID 不含空格，不设置固定颜色；
- 概念图优先表达业务角色和上下游，不混入函数名；
- 实现图可以出现文件、函数、数据库、事件、工具名或 Prompt，但必须先解释业务含义；
- 图不能替代定位、输入、输出和边界说明。

## 质量门

交付前确认：

- 先讲问题和本质，再讲代码；
- 有真实业务场景贯穿，不只是模块介绍；
- 实现篇每个关键阶段都包含业务解释和关键实现映射；
- 至少为关键输入输出提供 1–3 个短示例，例如 JSON、消息、状态、参数或代码片段；
- 分类先讲问题和例子，再映射实现；
- 每个流程环节都有输入输出；
- 通用概念、项目术语、教学类比和推断已区分；
- 函数名、变量名和文件路径服务解释，不喧宾夺主；
- 已说明代价和可迁移性；
- 表格只用于比较或查阅；
- 结论有官方文档或源码 anchors；
- 索引已更新；
- 后续追问可以只读索引、topic 和少量 anchors；
- 链接、JSON、Mermaid 和章节锚点有效。

# 增量索引与 Token 控制

## 索引职责

`01-index.json` 保存：

- 源码路径和版本；
- 模块与入口；
- 可追问特性；
- 已完成主题；
- 源码锚点；
- 自然语言问题路由；
- 源码变化后的失效状态。

它是后续分析的路由器，不是教学文档。

## 后续追问路由

### 问整体项目

读取 `00-overview.md` 和 `01-index.json`。除非需要验证，不读源码。

### 问已有主题

读取索引、对应 topic；只有问题超出文档时才读 2–6 个 anchors。

### 问新主题

读取索引和 candidate anchors，通过符号、调用关系与同义关键词扩展到约 5–15 个文件。完成后创建 topic 并更新索引。

### 问具体实现

从 topic 的最小源码路径进入，只查目标函数、直接调用链、相关数据类型、Prompt 和测试。不要重跑整体分析。

## Token 预算

### 整体分析

优先：

- 1 个根 README；
- 1–3 个架构或协议文档；
- 2–5 个入口；
- 少量核心类型；
- 关键模块 README。

官方文档已能回答的问题，不用读取多个实现重复证明。

### 特性分析

通常：

- 初始 anchors：2–4 个；
- 定向扩展：5–15 个文件；
- 正文源码路径：3–8 个；
- Prompt 档案：仅行为关键 Prompt。

当触发、输入、处理、输出、存储、消费、失败和安全边界都有证据时停止。

## 状态与刷新

Topic 状态：

- `current`：锚点存在且匹配记录版本；
- `needs-review`：源码变化，影响未知；
- `stale`：关键锚点消失或结论已过时。

有 git 时，将 diff 路径与 topic anchors 匹配，只刷新有交集的主题。核心协议或目录大规模重构时才刷新 overview。

无 git 时检查关键文件、符号和官方版本声明。不能确认时标记 `needs-review`。

## 回写规则

每次深潜更新：

- topic、Prompt、diagram 路径；
- 最小 anchors 和 symbols；
- `faq_routes`；
- 验证日期、revision 和状态；
- 新发现的同义关键词。

删除失效 anchor，不无限追加。一个主题只保留能重建主流程的最小证据集。

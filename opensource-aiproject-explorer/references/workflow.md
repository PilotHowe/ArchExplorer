# 分析工作流

## 1. 判断当前状态

不要默认这是首次分析。

1. **没有源码**：获取 URL 或本地路径。
2. **有源码、无索引**：执行首次整体分析。
3. **有索引、问整体架构**：先读 `00-overview.md` 和 `01-index.json`。
4. **有索引、问具体特性**：先查 `faq_routes`、`topics` 和 `feature_candidates`。

已有 topic 时，只按问题需要读取 2–6 个源码锚点。新 topic 从 candidate anchors 开始，通常扩展到 5–15 个相关文件。

## 2. 获取或识别源码

### 本地路径

1. 验证目录。
2. 用 README、构建文件或版本控制目录确认真正仓库根目录。
3. 注意压缩包可能形成 `<name>/<name>/` 两层目录。
4. 源码保持只读，分析产物写到独立 `projects/<name>/`。

### Git URL

1. 从 URL 推导项目名。
2. 检查目标父目录和同名目录。
3. 已有完整仓库时复用，不重复 clone。
4. 未存在时普通 clone；除非用户要求，不默认 shallow clone。
5. 记录 remote、branch、commit 或 tag。

认证失败时报告 blocker，不绕过权限。git 不可用时将 revision 记录为 `null`，不要伪造。

## 3. 首次扫描预算

第一轮只读：

- 根 README、许可证和语言/构建清单；
- 官方 architecture、design、protocol 文档；
- 2–5 个主入口；
- 顶层或核心模块 README；
- 解释一条典型请求路径所需的核心类型。

跳过 vendor、node_modules、target、dist 和大规模 fixtures。不要为了“完整”读取所有模块，也不要在尚未理解主流程时宽泛搜索所有 `agent`、`memory`。

### 大型仓库并行探索

可并行启动 2–3 个只读任务：

1. 产品定位、技术栈与官方术语；
2. 核心运行时与请求生命周期；
3. 执行、安全、存储和扩展能力。

每个任务返回结论、证据路径、关键符号、不确定项和候选特性。禁止无边界全仓总结。

## 4. 建立整体架构

先画用户请求主线：

```text
Client → Protocol/API → Agent/Runtime → Model
                        ↕
                  Tools/Environment
                        ↕
                    State/Storage
```

实际项目可以不同，但必须解释输入入口、任务状态、模型调用、外部动作、持久化和结果返回。

## 5. 生成整体分析

创建：

- `00-overview.md`：新手叙事，模块按概念分组；
- `01-index.json`：源码版本、模块、入口、候选特性、topic 和问题路由；
- `diagrams/architecture.mmd`：整体架构图源。

完整路径放索引，不要把总览变成 crate 或文件清单。

## 6. 执行特性深潜

先将用户问题拆为：

1. 概念为何出现；
2. 核心逻辑；
3. 上下游和分类；
4. 项目实现流程；
5. 各阶段输入输出；
6. 设计权衡；
7. 可迁移性。

搜索时组合用户术语、项目术语、机制术语、产物名称和入口事件，不依赖单个关键词。

优先形成以下调用链：

1. 触发入口；
2. orchestration；
3. 数据和状态模型；
4. Prompt 或策略；
5. 存储；
6. 读取或下游消费；
7. 测试或官方说明。

证据足以解释完整闭环后停止。

## 7. 回写 topic 与索引

创建 topic 后：

- 将 candidate 加入 `topics`；
- 记录 topic、Prompt、diagram；
- 添加最小 anchors 与 symbols；
- 添加自然语言 `faq_routes`；
- 记录 revision、验证日期和状态；
- 删除失效锚点，不无限追加。

## 8. 源码更新后的刷新

有 git 时，根据 revision diff 和 anchors 只刷新受影响 topic。没有 git 时，检查关键文件和符号是否仍存在；无法确认则标记 `needs-review`，不要以全仓重扫制造虚假确定性。

## 9. 停止条件

整体分析完成：

- 能解释请求生命周期；
- 核心概念边界清楚；
- 入口、核心、工具、状态有证据；
- 候选特性可路由。

特性分析完成：

- 能回答起源、本质和上下游；
- 分类先于代码；
- 每个阶段有定位、输入、处理、输出、边界；
- 能说明权衡和迁移；
- 后续追问无需重扫全仓。

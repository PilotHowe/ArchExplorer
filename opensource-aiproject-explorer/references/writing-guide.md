# 初学者文档写作规范

## 总原则

> 场景先于术语，概念先于实现，流程先于文件，业务动作先于函数名，示例连接抽象与源码，权衡先于结论，证据集中在最后。

文档必须让读者同时获得两层理解：

1. **业务逻辑层**：谁触发、谁负责、谁消费、状态如何流转、什么时候结束、失败归谁；
2. **实现落点层**：关键函数、变量、配置、工具、事件、数据结构和最小示例分别在哪里。

不要把“讲业务”写成泛泛而谈，也不要把“讲实现”写成文件和函数堆砌。推荐节奏是：

```text
真实场景 → 业务动作 → 输入输出 → 关键实现名 → 小示例 → 边界和代价 → 证据锚点
```

## 整体项目导读

按以下叙事展开：

1. 读完能回答什么；
2. 一句话定位；
3. 一个真实使用场景；
4. 整体设计思路；
5. 3–6 个核心概念；
6. 一次请求生命周期；
7. 按阅读顺序组织代码；
8. 值得深潜的关键特性；
9. 运行方式、常见混淆和源码阅读路径；
10. 证据与版本。

每个核心概念先通俗解释，再给项目正式定义，最后说明与相邻概念的区别。

## 特性深潜：概念篇

### 问题起源

依次说明：

1. 没有该机制时怎样工作；
2. 前身或朴素方案是什么；
3. 容量、成本、时间或可靠性变化后为何失效；
4. 当前概念带来了什么改变。

### 一句话本质

使用：

> `<上游输入>` 经过 `<核心选择或变换>`，成为 `<下游可利用的输出>`。

拆解输入、变换和输出，但不进入代码。

### 上下游地图

概念图同时展示所属大类、相邻机制、上游输入、下游消费者和内部分类。图后用文字带读者走一遍。

### 分类

每类按固定顺序：

```markdown
#### <类型名称>

**解决的问题**：它独立处理什么困难？

**例子**：在贯穿全文的业务场景中对应什么？

**边界**：与相邻类型有何区别？
```

所有类型讲完后，才建立“通用分类 → 项目代码”的映射。

## 特性深潜：实现篇

### 业务到实现的桥接

进入实现篇时，先用一小段把通用概念映射到项目术语：

- 业务角色对应哪些项目实体；
- 业务动作对应哪些工具、函数、事件或状态；
- 业务产物对应哪些参数、字段、文件、数据库行或消息；
- 哪些名称是项目原生术语，哪些是教学类比。

例如不要只写“调用 `spawn_agent`”，而要写清它在业务上是“主 Agent 创建一个子线程并投递第一张派工单”；然后再说明关键参数、返回路径和后续通信方式。

### 环节说明卡

每个 Phase、Stage、Pipeline 节点或读取层使用：

```markdown
#### <阶段名称>

**定位**：完整流程中的角色。

**解决的问题**：为何独立存在。

**输入**：数据、状态、策略和上游产物。

**处理**：
1. 关键动作；
2. 关键动作；
3. 关键动作。

**输出**：产物、状态和下游消费者。

**边界与代价**：不负责什么；为何并行、串行或按需。

**实现映射**：关键函数、结构体、变量、配置、事件名、工具名或文件路径。

**最小示例**：1–3 个短 JSON、消息、参数对象、状态流转、Prompt 片段或代码片段。
```

阶段结束后说明：为何不合并、前一阶段输出如何成为后一阶段输入、两者各自优化什么、失败与重试归谁。

### 示例写法

示例必须解释业务逻辑，不只是展示代码。优先使用短示例：

```json
{
  "task_name": "auth_test_investigation",
  "message": "只调查 auth 模块测试失败的根因，返回证据和建议修复点。"
}
```

示例后必须解释字段如何影响行为：

- 哪个字段是业务输入；
- 哪个字段决定寻址或身份；
- 哪个字段控制上下文、权限、成本或输出；
- 下游如何消费这个示例产生的结果。

代码片段只截取关键 5–15 行，除非用户明确要求完整源码。大段源码应留给证据锚点。

### 函数和变量解释

提到函数、变量或结构体时，必须说明它承担的业务责任：

- `xxx()` 负责把什么输入变成什么输出；
- `FooState` 表示哪类业务状态；
- `bar_id` 是身份、寻址、关联、去重还是权限边界；
- 某个事件表示业务流程进入哪个阶段。

不要只写“代码在 `foo.rs` 的 `bar()` 中实现”。应该写成：“主流程在 `bar()` 中把用户请求转换成执行任务，关键输入是 A，输出是 B，下游 C 会消费 B。”

### 分层读取

遇到 summary、handbook、detail、raw evidence 时，逐层说明：

- 内容粒度；
- 自动还是按需加载；
- token / I/O 成本；
- 适合回答的问题；
- 证据精度；
- 进入下一层的条件。

解释为何不能每次读最详细层、只保留短摘要，或让各层复制同一内容。

## 特性深潜：迁移篇

先抽象机制，例如：

```text
Capture → Select → Store → Retrieve → Inject → Feedback
```

再解释项目特有实现依赖的规模、生命周期和基础设施，给出更小替代方案、采用时机、最小闭环及效果指标。不能只写“可以复用”。

## 表格

表格只用于：

- 同维度方案对比；
- 配置查阅；
- 机器索引；
- 简短阶段交接摘要。

概念首次解释、因果关系、目录导读和完整流程使用正文。表格前说明比较目的，表格后给出结论。

## Mermaid

- 节点 ID 不含空格；
- 特殊标签使用双引号；
- 不设置固定颜色；
- 每图只表达一个关系；
- 概念图不混入函数名；
- 实现图才显示文件、数据库或 Prompt；
- 图后说明阅读顺序和关键分支。

如果同一主题既需要业务理解又需要源码理解，优先使用两张图：

- **业务图**：角色、动作、输入输出、通知和责任边界；
- **实现图**：函数、工具、事件、状态、文件或数据库如何串起来。

不要把两种图混成一张难读的大图。

## 术语与证据

区分项目原生术语、行业术语、教学类比和分析推断。正文只留最小源码路径，完整 anchors 放索引。可能变化的结论注明 revision 或验证日期。

## 自检

未读源码的人是否能回答：

1. 它解决什么问题？
2. 一句话本质是什么？
3. 上下游是什么？
4. 分类分别解决什么问题？
5. 各阶段输入输出是什么？
6. 为何分层读取？
7. 何时适合迁移？

准备读源码的人是否能回答：

1. 每个关键业务动作对应哪个函数、变量、工具、事件或数据结构？
2. 最小输入示例长什么样？
3. 输出或状态如何被下游消费？
4. 出错、超时、取消、权限不足时谁处理？
5. 如果只读 2–6 个源码锚点，应该从哪里开始？
